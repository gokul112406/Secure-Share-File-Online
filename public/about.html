🔒 How It Works

Zero-Knowledge
File Sharing

SecureShare uses AES-256-GCM — the same cipher used by governments and banks — to ensure your files are unreadable by anyone except the intended recipient.

🔐 Encryption Model

1. Key Generation

A cryptographically random 256-bit (32-byte) private key is generated using Node.js crypto.randomBytes(). This key is shown once and never stored.

2. AES-256-GCM Encryption

The file is encrypted with AES-256 in Galois/Counter Mode (GCM). A unique 96-bit IV is generated per upload. The auth tag prevents tampering.

3. Storage

Only the ciphertext, IV, and auth tag are stored in MongoDB. The private key is never written to any database or log.

4. Decryption

The receiver provides both keys. GCM authentication verifies integrity before decryption proceeds. A wrong key returns an error.

🛡️ Security Features

❓ Frequently Asked Questions

Can SecureShare read my files?

No. The private key is generated client-side during the upload request and displayed once. It is never sent to our database. We only store the encrypted ciphertext which is meaningless without the key.

What happens when a file expires?

MongoDB's TTL index automatically deletes expired documents. Any attempt to download after expiration returns a 410 Gone error.

What if I enter the wrong private key?

AES-256-GCM's authentication tag will fail verification. You'll receive a decryption error and no partial data is returned.

Is the file stored in plain text during transit?

No. The file is encrypted on the server immediately upon receipt using multer's memory storage. The raw bytes never touch the disk.
